# 数据库迁移规范

## 1. 目的

本文档定义本项目数据库迁移的唯一规范。

本项目不使用外部迁移框架，不使用自动生成迁移的工具，不依赖数据库内的迁移记录表，也不依赖独立的 manifest 文件。数据库结构变更完全由人工编写 SQL 迁移文件、人工评审、人工执行、人工留痕。

凡是涉及以下内容的任务，都必须先阅读本文档：

- SQLAlchemy 模型字段变更
- 新增或删除表
- 新增或删除列
- 索引、唯一约束、外键、默认值、可空性变更
- 数据回填、数据修复、历史脏数据清洗
- 任何需要手写 SQL 的结构变更

## 2. 适用范围

本文档适用于本仓库内所有数据库相关改动，包括但不限于：

- `backend/app/models.py`
- 后续新增的模型文件
- 应用运行所依赖的外部数据库表，例如 Agno 使用的 `agent_sessions`
- `backend/migrations/versions/` 下的所有 SQL 文件
- 发布前后的数据库结构检查

## 3. 核心法则

以下规则是硬规则，不得突破：

1. 历史迁移文件一旦提交到仓库，永不修改。
2. 历史迁移文件一旦被任何共享环境执行，永不删除。
3. 任何数据库结构变更都必须通过新增迁移文件完成，不允许靠修改历史迁移文件表达新需求。
4. 正式环境数据库迁移只能由人工执行，禁止依赖应用启动流程自动修表。
5. 模型变更如果影响数据库结构，必须同步提交迁移文件；模型改了但没有迁移文件，视为任务未完成。
6. 迁移文件以生产数据库方言为准。本项目生产数据库为 MySQL，因此迁移 SQL 默认按 MySQL 编写。
7. SQLite 只可作为本地开发便利，不作为迁移正确性的最终依据。
8. 一个迁移文件只做一组高度相关的变更，不允许把多个无关需求混在同一个文件里。
9. 迁移失败后，不允许通过修改原迁移文件补救，只能回滚数据库备份，或新增修复迁移文件。
10. 任何会导致数据丢失、锁表、重建大表、长期阻塞的操作，必须在迁移文件头部显式标记风险。
11. 外部库运行时依赖的表如果在生产数据库中使用，也必须纳入人工迁移文件，不允许由运行时代码在生产环境自动建表。

## 4. 目录约定

所有迁移文件统一存放在以下目录：

`backend/migrations/versions/`

只允许在这个目录中按时间顺序追加新的 `.sql` 文件，不允许在其他位置散落迁移脚本。

## 5. 文件命名规范

迁移文件必须使用以下格式命名：

`YYYYMMDDHHMMSS__short_description.sql`

示例：

`20260508213045__add_volume_id_to_chapters.sql`

具体要求如下：

- 时间戳必须精确到秒，使用北京时间书写。
- `short_description` 使用简短英文描述，推荐小写字母和下划线。
- 文件名中不要出现空格。
- 文件名中不要出现中文。
- 文件名一旦提交，禁止修改。
- 新文件必须按时间顺序追加，不能插队伪造更早版本。

## 6. 什么时候必须新增迁移文件

只要出现下列任一情况，就必须新增迁移文件：

- 新建表
- 删除表
- 重命名表
- 新增列
- 删除列
- 重命名列
- 修改列类型
- 修改列长度
- 修改列默认值
- 修改列是否允许为空
- 新增索引
- 删除索引
- 新增唯一约束
- 删除唯一约束
- 新增外键
- 删除外键
- 调整主键
- 增加枚举值或状态值并要求数据同步
- 数据回填
- 历史数据修复
- 为了满足新约束而进行的数据清洗

以下情况通常不需要新增迁移文件：

- 仅修改 Python 侧业务逻辑，且不影响数据库结构与历史数据
- 仅修改查询条件、排序逻辑、接口返回结构，且不影响数据库结构

如有争议，按“需要迁移文件”处理，不做侥幸判断。

## 7. 写迁移前必须做的事

在新建迁移文件之前，必须按顺序完成以下动作：

1. 阅读本文档。
2. 阅读 `backend/migrations/versions/` 中最新的相关迁移文件。
3. 如果当前变更和更早的结构演进有关，继续追溯相关历史迁移文件，而不是只看最新一个文件。
4. 检查当前目标数据库结构，确认真实状态与预期状态。
5. 明确本次变更是结构迁移、数据迁移，还是二者混合。
6. 判断本次改动是否可以一次完成，还是必须拆成多步迁移。
7. 判断本次变更是否存在数据丢失风险、锁表风险、长事务风险或兼容性风险。

## 8. 迁移文件头部模板

每个迁移文件开头都必须包含完整注释，格式统一如下：

```sql
-- Version: 20260508213045
-- Title: add_volume_id_to_chapters
-- Author: your_name
-- Created At: 2026-05-08 21:30:45 +08:00
-- Target: mysql8
-- Type: schema
-- Risk: medium
-- Reversible: no
-- Summary:
-- 1. Add chapters.volume_id.
-- 2. Add index on chapters.volume_id.
-- 3. Change unique constraint scope from work to volume.
--
-- Pre-Checks:
-- 1. Confirm a full database backup has completed.
-- 2. Confirm no existing rows will violate the new unique constraint.
--
-- Execution Notes:
-- 1. Execute during a low-traffic window if the table is large.
-- 2. Do not edit this file after it is committed.
--
-- Post-Checks:
-- 1. SHOW CREATE TABLE chapters;
-- 2. Verify the application can create, update, and query chapters normally.
```

要求如下：

- `Version` 必须与文件名中的时间戳完全一致。
- `Title` 必须与文件名中的描述部分语义一致。
- `Author` 写真实责任人。
- `Created At` 使用北京时间。
- `Target` 默认写 `mysql8`。
- `Type` 只能写 `schema`、`data`、`mixed` 三者之一。
- `Risk` 只能写 `low`、`medium`、`high` 三者之一。
- `Reversible` 默认写 `no`。本项目不依赖回滚脚本表达回滚能力。
- `Summary` 必须用简洁条目说明本次迁移做了什么。
- `Pre-Checks` 必须写执行前要确认的条件。
- `Execution Notes` 必须写执行注意事项。
- `Post-Checks` 必须写执行后检查项。

## 9. SQL 编写规范

### 9.1 总体要求

- 迁移 SQL 必须由人工编写。
- 迁移 SQL 必须可读、可审查、可解释。
- 迁移 SQL 必须以生产数据库方言为准。
- 迁移 SQL 必须让风险显式暴露，不能为了“重复执行不报错”而掩盖问题。

### 9.2 明确禁止

以下做法禁止使用：

- 修改历史迁移文件
- 删除历史迁移文件
- 在迁移文件中混入多个无关需求
- 为了省事直接重建整张表，除非确实没有更安全的办法
- 为了兼容重复执行而滥用 `IF EXISTS`、`IF NOT EXISTS`
- 在没有数据检查的前提下直接收紧唯一约束、非空约束
- 在不了解现有数据分布的情况下直接修改列类型
- 用 SQLite 语法代替 MySQL 语法作为正式迁移依据

### 9.3 推荐写法

- 一条迁移只表达一次清晰意图。
- 先在注释中写清 pre-check，再写正式 SQL，再在注释中写清 post-check。
- 结构变更与大规模数据回填尽量分开，不要在同一个迁移里做过多动作。
- 对于大表或高风险变更，优先拆成多次迁移，降低单次变更范围。
- 对于需要人工观察结果的步骤，在注释中写清必须确认的结果。

### 9.4 关于幂等性

本项目迁移文件的设计目标不是“可无限重复执行”，而是“在明确顺序下被人工执行一次”。

因此：

- 不要为了制造伪幂等而隐藏错误。
- 不要把真正的执行失败包装成“看起来成功”。
- 如果某条 SQL 执行失败，应当先停下来查明原因，而不是继续写更多容错语句掩盖现场。

## 10. 高风险变更的处理原则

### 10.1 重命名表或列

重命名表、重命名列在很多情况下不应被视为单步小改动。

默认做法：

1. 先新增新结构。
2. 回填或迁移数据。
3. 应用切换到新结构。
4. 验证稳定。
5. 再删除旧结构。

除非你非常确定风险可控，否则不要把“rename”当成一步完成的安全操作。

### 10.2 新增非空约束

对已有数据的表新增 `NOT NULL` 时，默认流程为：

1. 先新增可空列，或保留原列可空。
2. 先回填历史数据。
3. 验证所有历史行都满足要求。
4. 再通过后续迁移收紧为 `NOT NULL`。

### 10.3 新增唯一约束

新增唯一约束前，必须先确认历史数据中不存在重复值。

如果存在重复值，必须先通过独立迁移处理脏数据，再新增约束。

### 10.4 删除列或删除表

删除列、删除表是高风险操作，默认不应与“应用刚切新版本”同时发生。

建议流程：

1. 先让应用停止依赖旧结构。
2. 观察一段时间，确认没有读写旧结构。
3. 再新增清理迁移删除旧结构。

### 10.5 大表变更

如果目标表很大，或变更可能导致长时间锁表，必须在迁移文件头部明确写出：

- 是否需要维护窗口
- 是否需要流量切换
- 是否需要先做备份
- 是否需要人工分批执行

## 11. expand-contract 原则

凡是涉及线上兼容风险的变更，优先采用 expand-contract，而不是一步到位。

标准思路如下：

### 第一步：expand

- 新增列、新增索引、新增新表
- 保持旧结构继续可用
- 允许新旧结构并存

### 第二步：切换

- 应用切换到新结构
- 必要时完成数据回填
- 观察并验证业务稳定

### 第三步：contract

- 删除旧列、旧索引、旧约束、旧表
- 收紧非空或唯一约束

如果某个改动会影响线上读写兼容性，就不要试图一步做完。

## 12. PR 约束

只要 PR 中出现以下内容之一，就必须检查是否同时提交了迁移文件：

- 模型结构变更
- 新增或修改 SQL
- 约束、索引、外键变更
- 数据回填逻辑变更

PR 审查时必须检查：

- 是否新增了迁移文件
- 文件名是否符合规范
- 头部注释是否完整
- 是否修改了历史迁移文件
- SQL 是否与模型改动一致
- 是否写清 pre-check 和 post-check
- 是否存在数据丢失风险
- 是否应该拆分为多步迁移

如果模型结构改了但没有迁移文件，PR 不应通过。

## 13. 正式环境执行规范

正式环境数据库迁移必须由人工执行。

执行前必须完成以下动作：

1. 明确本次上线需要执行哪些迁移文件。
2. 确认执行顺序正确。
3. 先看每个迁移文件头部的 `Pre-Checks`。
4. 完成数据库备份或确认已有可用备份。
5. 确认当前数据库状态与预期一致。
6. 评估是否需要维护窗口。

执行时必须遵守以下规则：

- 严格按文件时间顺序执行。
- 不允许跳过中间版本。
- 不允许手工改写 SQL 后再执行。
- 不允许临时修改仓库内迁移文件内容。
- 若执行失败，立即停止，不允许在现场继续“边试边改”原文件。

执行后必须完成以下动作：

1. 执行 `Post-Checks` 中列出的检查。
2. 验证核心业务路径是否正常。
3. 在发布记录、工单系统或运维记录中登记本次执行结果。

登记内容至少包括：

- 环境名称
- 执行时间
- 执行人
- 执行的迁移文件名
- 是否成功
- 失败原因或补充说明

## 14. 失败处理原则

如果迁移执行失败，必须遵循以下原则：

1. 先停止，不继续扩大变更范围。
2. 保留失败现场，先判断失败位置和数据库当前状态。
3. 不允许回仓库修改已提交的历史迁移文件。
4. 根据实际情况选择：
   - 恢复数据库备份
   - 新增修复迁移文件
5. 修复方案必须重新评审后再执行。

禁止做法：

- 直接覆盖旧迁移文件内容
- 假装失败没有发生
- 在未记录现场状态的情况下继续执行后续迁移

## 15. 标准迁移文件模板

以下模板可直接复制使用：

```sql
-- Version: 20260508213045
-- Title: change_this_to_real_title
-- Author: your_name
-- Created At: 2026-05-08 21:30:45 +08:00
-- Target: mysql8
-- Type: schema
-- Risk: low
-- Reversible: no
-- Summary:
-- 1. Describe the primary schema change.
-- 2. Describe the secondary related change if needed.
--
-- Pre-Checks:
-- 1. Describe what must be verified before execution.
-- 2. Describe any data validation or backup requirement.
--
-- Execution Notes:
-- 1. Describe any operational caution.
-- 2. State whether a maintenance window is required.
--
-- Post-Checks:
-- 1. Describe what must be validated after execution.
-- 2. Describe what business path should be smoke-tested.

-- Write executable SQL below this line.
```

## 16. 示例

下面是一个简单示例：

```sql
-- Version: 20260508213045
-- Title: add_last_login_ip_to_users
-- Author: alice
-- Created At: 2026-05-08 21:30:45 +08:00
-- Target: mysql8
-- Type: schema
-- Risk: low
-- Reversible: no
-- Summary:
-- 1. Add users.last_login_ip.
-- 2. Add an index for users.last_login_ip.
--
-- Pre-Checks:
-- 1. Confirm the users table exists.
-- 2. Confirm no application code depends on a different column name.
--
-- Execution Notes:
-- 1. Execute during normal release flow.
-- 2. Do not edit this file after commit.
--
-- Post-Checks:
-- 1. SHOW CREATE TABLE users;
-- 2. Verify login flow still works.

ALTER TABLE users
ADD COLUMN last_login_ip VARCHAR(45) NULL;

CREATE INDEX ix_users_last_login_ip ON users (last_login_ip);
```

## 17. 与本项目当前现状的关系

当前仓库中存在本地自动建表与轻量修补逻辑，但这不应被视为正式环境结构迁移方案。

从本规范生效开始，应遵循以下原则：

- 新的正式数据库结构变更，不得依赖“应用启动时自动修表”完成。
- 新的模型结构变更，必须配套新增 SQL 迁移文件。
- 新的数据库发布，必须由人工按迁移文件顺序执行。

若旧逻辑仍存在，只代表历史兼容，不代表未来可继续依赖。

首个迁移文件是当前模型和运行时依赖表的初始基线，面向空 MySQL 数据库执行。已经由旧逻辑创建过表的环境，需要人工确认现有结构与基线一致，并在发布记录、工单系统或运维记录中登记为已完成基线化。

## 18. 一句话总结

本项目数据库迁移规则只有一句话：

**任何数据库结构变更，都必须以新增、不可变、人工编写、人工执行的 SQL 迁移文件来完成。**
