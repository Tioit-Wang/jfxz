# 金番写作 MVP 技术选型报告

**版本**：V1.2  
**文档类型**：技术选型报告

## 一、文档目标

本报告用于确认金番写作 MVP 的关键技术决策，覆盖：

- 用户端与管理端的技术组织方式
- 前端框架与 UI 方案
- 编辑器与输入框技术路线
- 后端框架与流式通信方案
- Agent 框架接入方案
- 模型来源与模型计费方案
- 数据持久化与计费数据方案

本文件只说明技术决策，不重复需求文档中的产品交互细节。

## 三、系统结构

系统分为两部分：

- **用户端**：作者使用的创作工作台
- **管理端**：内部后台，用于用户、模型、套餐、订单订阅、积分流水、会话和系统配置管理

推荐采用同一套 Next.js 工程承载两端，通过路由分组隔离：

- 用户端：`/works`、`/profile`、`/billing`
- 管理端：`/admin/users`、`/admin/models`、`/admin/products`、`/admin/orders`、`/admin/subscriptions`、`/admin/point-transactions`、`/admin/sessions`、`/admin/configs`

后端同样按接口分组：

- 用户端 API
- 管理端 API
- 鉴权中间层按 `role` 区分 `user` / `admin`

## 四、前端选型

### 4.1 Next.js

选择 Next.js 的原因：

1. React 生态成熟，适合快速搭建工作台类产品。
2. 适合一个工程承载用户端与管理端两套路由。
3. 与 Tailwind CSS、shadcn/ui、TipTap 集成成本低。
4. 适合承载 SSE 流式界面和复杂状态。

### 4.2 shadcn/ui + Tailwind CSS

选择原因：

- 组件源码可控，便于统一用户端和管理端风格
- Tailwind CSS 适合快速完成工作台、后台列表、表单和弹层
- 对 AI 辅助开发更友好

## 五、编辑器与输入框方案

### 5.1 编辑器：TipTap

选择原因：

1. 可通过最小节点集实现纯文本写作。
2. 基于 ProseMirror，具备成熟的装饰和扩展能力。
3. 适合实现 AI 高亮、提示气泡和建议覆盖层。

### 5.2 对话输入框：TipTap Mention Extension

选择原因：

1. 与编辑器统一技术栈，减少维护成本。
2. 易于支持 `@章节`、`@角色`、`@设定` 的结构化引用。
3. 易于实现候选浮窗、键盘导航和标签渲染。

## 六、后端选型

### 6.1 FastAPI + Python

选择原因：

1. 适合承载异步接口、模型调用和流式返回。
2. Pydantic、依赖注入和 OpenAPI 能降低接口开发成本。
3. Python 在 LLM、Agent 和支付/订阅集成周边上生态成熟。

### 6.2 SSE 作为流式输出协议

对话场景采用 SSE，原因如下：

- 实现简单，适合单向 token 流输出
- 与 FastAPI `StreamingResponse` 配合直接
- 对当前 MVP 的智能对话面板已经足够

当前不优先采用 WebSocket，原因是：

- 当前核心需求是服务端持续输出，而不是复杂双向实时协作
- SSE 实现成本更低，调试更简单

## 七、数据库与数据分层

### 7.1 PostgreSQL

选择 PostgreSQL 的原因：

- 结构化数据建模能力强
- 可统一承载创作数据、账户数据和商业化数据
- 可同时承接 Agno Session Storage 的持久化

### 7.2 数据分层原则

数据库层分为五类数据：

- **账号与权限数据**：`users`
- **系统配置数据**：`global_configs`
- **创作数据**：`works`、`characters`、`settings`、`chapters`
- **对话数据**：`chat_sessions`、`agent_sessions`
- **AI 模型数据**：`ai_models`
- **商业化与支付数据**：`plans`、`credit_packs`、`billing_orders`、`payment_records`、`payment_notify_logs`、`user_subscriptions`、`point_accounts`、`point_transactions`

详细字段定义见 `数据库设计文档.md`。

## 八、认证与权限模型

MVP 阶段采用统一账号模型：

- 用户与管理员共用 `users` 表
- 通过 `role` 区分 `user` / `admin`
- 通过 `status` 控制 `active` / `disabled`

这样可以避免额外拆 `admin_users`，降低实现复杂度。

## 九、套餐与计费模型

MVP 阶段采用“套餐 + 积分权益包 + 积分”的方式：

- **套餐** = VIP 每日积分计划 + 附带积分权益包
- **积分权益包** 可以单独售卖，永久有效，直接叠加余额
- 用户侧按积分消费，用户流水只展示时间、积分变动或消耗积分、关联作品

技术实现建议：

- 模型定义、评分、上下文参数和计费倍率存于 `ai_models`
- 套餐定义存于 `plans`
- 独立积分权益包定义存于 `credit_packs`
- 订单统一记录在 `billing_orders`
- 套餐订阅记录在 `user_subscriptions`
- 当前积分余额记录在 `point_accounts`
- 每次积分变动记录在 `point_transactions`
- AI 调用前按当前模型最大输出和本次上下文估算最高积分消耗，余额不足时拒绝调用
- AI 调用成功后按 OpenAI-compatible usage 写入积分消费流水

这样既能支撑订阅和续费顺延，又能支撑单独售卖积分权益包。

### 9.1 Token 计费基准

模型调用统一口径见 `管理端需求文档/模型管理需求文档.md`，包括供应商地址、请求协议、API key 管理、调用 ID 和 usage 解析规则。

计费基准：

- 基准价：`10 元 / 100万 token`
- 积分面值：`1 积分 = 0.1 元`
- 基准积分单价：`100 积分 / 100万 token`
- 扣费积分保留 2 位小数
- 模型倍率保留 2 位小数并向上取整

扣费使用 `prompt_cache_hit_tokens`、`prompt_cache_miss_tokens` 和 `completion_tokens`。`reasoning_tokens` 已包含在 `completion_tokens` 中，不重复计费。

## 十、支付接入模型

MVP 阶段当前只接入一个支付渠道：

- **支付宝当面付（扫码支付）**

技术实现原则：

- 订单先写入 `billing_orders`
- 服务端调用支付宝预下单接口生成二维码
- 支付渠道记录写入 `payment_records`
- 异步通知日志写入 `payment_notify_logs`
- 支付配置统一存于 `global_configs`

当前推荐的支付配置项放在 `payment.alipay_f2f` 分组下，由管理端系统配置模块维护。

详细支付流程、状态与数据关系见 `支付接入需求文档.md`。

## 十一、Agent 框架选型

### 10.1 选择 Agno

当前对话功能的特点是：

- 单 Agent 即可完成主要工作
- 需要查询作品、角色、设定、章节等上下文
- 需要多轮对话和会话持久化
- 需要兼顾流式返回和后续优化能力

选择 Agno 的原因：

1. 能力覆盖当前单 Agent 场景。
2. 官方提供 Session Storage，适合会话型产品。
3. 与 Python 服务端集成自然。
4. 后续可继续扩展 history、summary、memory 等能力。

### 10.2 当前采用的 Agno 能力

MVP 阶段采用：

- Agent
- Tools
- Session Storage
- History to Context

MVP 阶段暂不默认采用：

- User Memory
- Cross-session Search
- 更复杂的多 Agent 编排

## 十三、开发优先级建议

建议按以下顺序推进：

1. 建立 `users`、`global_configs`、创作数据表、套餐与积分相关表
2. 打通认证与角色权限
3. 打通支付宝当面付预下单、通知和查单链路
4. 打通 Agno Session Storage
5. 完成对话 SSE 链路
6. 完成模型管理和模型选择配置
7. 完成套餐购买、支付成功发放权益和积分扣减闭环
8. 完成积分流水只读查看
9. 完成管理端最小后台

## 十四、风险与应对

| 风险                        | 等级 | 应对策略                                     |
| :-------------------------- | :--- | :------------------------------------------- |
| TipTap 自定义交互复杂       | 中   | 限制首期能力，只实现必要高亮与覆盖层         |
| Agno 版本变动               | 中   | 锁定版本，Agent 能力封装在服务层             |
| 套餐与积分规则过于复杂      | 中   | 首期只做 VIP 每日积分 + 附带积分权益包 + 独立积分权益包 |
| 模型倍率配置错误            | 中   | 管理端支持成本价和目标加价率快速生成倍率，并保留历史倍率快照 |
| 支付回调与查单并发导致重复发放 | 中 | 订单发放逻辑做幂等控制                       |
| 长会话 token 成本上升       | 中   | 首期仅注入最近历史，后续再开启摘要           |
| 长章节内容超出模型上下文    | 高   | 分段分析正文，必要时切换更大上下文模型       |

## 十五、结论

当前技术选型满足以下目标：

- 能快速完成用户端 MVP
- 能以较低成本补齐管理端最小后台
- 能支撑 Agent 型多轮对话
- 能支持套餐、积分权益包、支付和积分体系
- 能支持模型管理、模型选择、usage 扣费和积分流水追溯
- 能与当前作品、角色、设定、章节模型保持一致

该方案适合作为第一期开发基线。
